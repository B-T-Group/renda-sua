# Notifications Module

This module provides email notifications for the Rendasua order management system using [Resend](https://resend.com/) transactional templates.

## Overview

The notifications system automatically sends email notifications to clients, businesses, and agents whenever:

- An order is created
- An order status changes

## Features

- **Resend integration**: Sends via Resend `emails.send` with published template IDs
- **Template-based emails**: HTML sources live under `templates/en` and `templates/fr`; sync with `npm run sync:resend-templates`
- **Multi-recipient Support**: Sends appropriate notifications to different user types
- **Error Handling**: Graceful failure handling that doesn't break order operations
- **Configurable**: Easy to configure via environment variables

## Configuration

### Environment Variables

Add these environment variables to your configuration:

```bash
# Resend (local + production when key is set)
RESEND_API_KEY=re_xxxxxxxx
# Optional override; default in code is Rendasua <noreply@rendasua.com>
RESEND_FROM_EMAIL=Rendasua <noreply@rendasua.com>

# Optional: disable order-status emails without removing the API key
ORDER_STATUS_NOTIFICATIONS_ENABLED=true
```

### Resend setup

1. Create a Resend account and API key with sending permission.
2. From the repo root, run `npm run sync:resend-templates` with `RESEND_API_KEY` set. This creates/updates published templates and writes UUIDs to `resend-template-ids.json`.
3. Deploy the backend with the same `RESEND_API_KEY` (and optional `RESEND_FROM_EMAIL`) in environment or secrets.

### Template IDs

Runtime template UUIDs are loaded from `resend-template-ids.json` (generated by the sync script). Until IDs exist for a key, that email is skipped with a warning.

## Usage

### Automatic Notifications

Notifications are automatically sent when:

1. **Order Creation**: When `OrdersService.createOrder()` is called (still synchronous from the API; may also enqueue `order.created` for Lambda-side work).
2. **Status Changes**: When `OrderStatusService.updateOrderStatus()` succeeds, the API enqueues an **`order.status.updated`** message on the FIFO SQS queue (`ORDER_STATUS_QUEUE_URL`) with `orderId`, `previousStatus`, `status` (new status), optional `actorUserId` (Hasura `users.id` of the caller), and `timestamp`. The **order-status-handler** Lambda processes the message (e.g. proximity row for `ready_for_pickup`) and then calls the backend **`POST /api/notifications/internal/order-status-change`** with the same `X-Rendasua-Internal-Key` used for internal SMS. Nest loads order payload and runs `NotificationsService.sendOrderStatusChangeNotifications`, respecting `ORDER_STATUS_NOTIFICATIONS_ENABLED` / `notification.orderStatusChangeEnabled`. This keeps Resend/Expo/web-push logic in one place and avoids blocking the status API on outbound providers.

The acting user is omitted from email/push recipient lists when their `users.id` matches `actorUserId` (e.g. assigned agent after pickup).

**Order status push (in Nest, after Lambda invokes internal route):** Push is sent per recipient using **`users.id`** when available (Expo tokens from `mobile_push_tokens` and/or web `push_subscriptions` with VAPID). A recipient **does not need an email** on the payload to receive push if `clientUserId` / `businessUserId` / `assignedAgentUserId` is present on the loaded `NotificationData`.

**Push-only channel:** When **`push.enabled`** and Hasura shows that user has at least one valid Expo token or at least one web subscription (with VAPID configured), **Resend order-status emails and client Orange SMS** for that user are skipped for that event; only push is used.

**Observability:** Each successful web push subscription send and each successful Expo chunk logs a line (`Push sent channel=web|expo userId=… orderId=…`). If valid Expo tokens exist but the Expo SDK client cannot be created while push is enabled, a **warning** is logged and Expo sends are skipped for that user (web push may still run).

**Pay-by-link / payment confirmation**: When `OrdersService` moves an order from `pending_payment` to `pending` after payment, it enqueues the same `order.status.updated` message with `previousStatus: pending_payment` and `actorUserId` set to the client’s user id.

**Requirements for Lambda → Nest**: Set `BACKEND_INTERNAL_API_BASE_URL` and `NOTIFICATIONS_INTERNAL_API_KEY` on the **order-status-handler** Lambda (CDK passes base URL; key via `process.env.NOTIFICATIONS_INTERNAL_API_KEY` at synth/deploy or AWS console). These must match Nest `NOTIFICATIONS_INTERNAL_API_KEY`.

After changing Python under `apps/cdk/src/core-packages/`, rebuild and publish the **core-packages** Lambda layer zip used by CDK (`core-packages-layer.zip`) so `nest_order_status_notifications_client` is included.

## Integration Points

### Orders Service

The `OrdersService.createOrder()` method automatically sends notifications to:

- Client (order confirmation)
- Business (new order alert)

### Order Status Service

After a successful status mutation, **`OrderStatusService.updateOrderStatus()`** enqueues **`order.status.updated`** for **all** transitions including **`complete`** (so client-driven `delivered → complete` still triggers notifications). Customer-facing emails and push are delivered asynchronously via Lambda + internal Nest route as described above.

Recipient matrix for each **new** status (before actor exclusion):

- **confirmed**: Client + Business
- **preparing**: Client + Business
- **ready_for_pickup**: Client + Business
- **assigned_to_agent**: Agent only
- **picked_up**: Client + Business + Agent
- **in_transit**: Client + Business + Agent
- **out_for_delivery**: Client + Business + Agent
- **delivered**: Client + Business + Agent
- **complete**: Client + Business + Agent
- **cancelled**: Client + Business + Agent (if assigned)
- **failed**: Client + Business + Agent (if assigned)
- **refunded**: Client + Business

## Error Handling

- Status mutations do not wait on Resend, Expo, or web-push; failures are logged on the Lambda / internal route and can be retried via SQS visibility timeout.
- If the internal notify call returns `{ success: false }`, the Lambda records `nest_notify_detail` in logs; the order row is already updated.
- The system gracefully handles missing Resend template IDs (skips with a warning).

## Template Variables

All templates support the following variables:

### Common Variables

- `orderId` - Order UUID
- `orderNumber` - Human-readable order number
- `orderStatus` - Current order status
- `orderItems` - Array of order items
- `subtotal` - Order subtotal
- `deliveryFee` - Delivery fee
- `taxAmount` - Tax amount
- `totalAmount` - Total order amount
- `currency` - Currency code
- `deliveryAddress` - Delivery address
- `estimatedDeliveryTime` - Estimated delivery time
- `specialInstructions` - Special delivery instructions
- `currentYear` - Current year for copyright

### User-specific Variables

- `recipientName` - Name of the email recipient
- `clientName` - Client's name
- `businessName` - Business name
- `agentName` - Agent's name (when applicable)

## Development

### Adding New Templates

1. Create HTML template in `templates/` directory
2. Add template ID to `templateIds` object in service
3. Update recipient logic in `getRecipientsForStatus()` method
4. Test with notification controller endpoints

### Template Styling

Templates use inline CSS for maximum email client compatibility. Key styling guidelines:

- Use inline styles
- Test across different email clients
- Keep responsive design simple
- Use web-safe fonts
- Include fallback colors

## Monitoring

Monitor notification delivery through:

- SendGrid dashboard for delivery statistics

- SendGrid webhook events for detailed delivery status

- API keys are stored securely in environment variables
- Email addresses are validated before sending
- Templates are sanitized to prevent injection attacks
